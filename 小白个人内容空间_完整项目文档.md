# 小白个人内容空间：项目交付包

这是交给 Codex 使用的产品与技术文档包。

## 项目目标

开发一个真正“打开就能写”的个人内容网站：

- 公开端用于展示自我、心得、随笔、项目和知识理解；
- 私有写作台用于登录、写作、插入图片、保存草稿和发布；
- 日常写作不需要进入 GitHub，也不需要手动创建 Markdown、Commit 或部署；
- 代码仍保存在 GitHub 仓库中，便于 Codex 持续开发和版本管理；
- 内容保存在自己的数据库和图片存储中，并支持导出，避免平台锁定。

## 推荐开发方式

不要让 Codex 一次性完成全部功能。

```text
阶段 0：初始化项目与基础规范
阶段 1：公开端静态页面与视觉骨架
阶段 2：Supabase 数据库、认证和权限
```

每个阶段结束后：

1. 查看 Codex 修改的文件；
2. 运行构建与检查；
3. 对照验收清单；
4. 确认后再进入下一阶段。

# 00 产品愿景

## 1. 产品是什么

“小白的个人内容空间”是一套由用户自己拥有的个人发布系统。

它不是传统博客模板、需要每次进入 GitHub 写 Markdown 的静态站、学术主页、只展示最终结果的作品集，也不是面向大众运营的内容平台。

它是：

> 一个能够随时打开、直接写作、插入图片、保存草稿并发布的个人空间，用来长期保存项目实践、学习心得、随笔思考和知识理解。

## 2. 核心问题

### 静态博客

优点是代码和内容都在仓库中，但日常发布步骤过多：

```text
进入仓库 → 新建文件 → 写 Front Matter → Commit → 等待部署
```

它适合维护代码，不适合低阻力写作。

### Notion 类工具

写作简单，但页面风格、信息架构和数据控制受平台限制，难以形成完全属于自己的产品。

### 通用博客后台

写作方便，但作品、心得、知识理解和个人成长通常被压缩成同一种“文章”，无法体现用户自己的表达框架。

## 3. 产品价值

### 低阻力

```text
打开网站 → 登录 → 选择类型 → 写作 → 插图 → 发布
```

### 有结构

内容分成：心得、随笔、项目、理解。

### 能表达过程

不仅展示“做成了什么”，也展示原来如何理解、实际发生了什么、遇到什么问题、为什么调整，以及现在如何理解。

### 内容属于自己

- 代码在 GitHub；
- 文章在自己的数据库；
- 图片在自己的 Storage；
- 可导出为 Markdown 或 JSON；
- 后续可以迁移到其他系统。

## 4. 核心用户

第一阶段只有一个用户：网站所有者本人。

公开访客只能阅读已发布内容，不设计公众注册和多人协作。

## 5. 产品原则

- 内容优先：设计服务于表达。
- 写作优先：任何功能都不能显著增加开始写作的步骤。
- 过程真实：不把未完成项目包装成完成品。
- 渐进复杂：先完成稳定闭环，再增加高级功能。
- 可迁移：任何时候都能导出内容。
- 公开与私密分离：草稿、后台和媒体管理必须受保护。

## 6. 成功标准

第一版成功不是“功能很多”，而是用户能连续完成：

```text
登录
→ 新建一篇心得
→ 插入两张图片
→ 自动保存
→ 点击发布
→ 在公开页面立即看到内容
→ 手机端正常阅读
```

# 01 V0.1 产品需求文档

## 1. V0.1 目标

完成一个可部署、可登录、可写作、可插图、可发布的个人内容系统。

必须形成完整闭环：

```text
登录后台 → 新建内容 → 选择类型 → 使用模板或空白内容
→ 编辑文字与图片 → 保存草稿 → 发布 → 公开端显示 → 后续重新编辑
```

## 2. 用户角色

### 访客

可以浏览首页、内容空间、分类、已发布内容、项目和关于页面，并搜索公开内容。

不能访问后台、查看草稿、修改内容或上传文件。

### 管理员

只有网站所有者一个人，可以登录、新建、编辑、发布、撤回、上传图片、设置精选、删除到回收站、恢复和导出数据。

## 3. 内容类型

| 类型值 | 中文名称 | 用途 |
|---|---|---|
| `reflection` | 心得 | 学习、实践后的认知变化 |
| `essay` | 随笔 | 不要求完整论证的个人思考 |
| `project` | 项目 | 图文展示项目过程与结果 |
| `understanding` | 理解 | 用自己的语言解释知识 |

## 4. 内容状态

| 状态值 | 中文 | 行为 |
|---|---|---|
| `draft` | 草稿 | 仅管理员可见 |
| `published` | 已发布 | 公开端可见 |
| `archived` | 已归档 | 后台保留，公开端默认隐藏 |

删除使用 `deleted_at` 软删除。

## 5. 公开端页面

### 5.1 首页 `/`

模块顺序：顶部导航、Hero、个人介绍、当前关注、精选内容、精选项目、最近更新、页脚。

Hero 包含网站名、一句话定位、简短说明，以及“进入内容空间”和“查看项目”两个按钮。

### 5.2 内容空间 `/space`

显示四个分类组件：心得、随笔、项目、理解。

每个分类卡片显示名称、说明、已发布数量、最近更新时间、最近一条内容和进入按钮。

### 5.3 分类页 `/space/[type]`

- 分类介绍；
- 按发布时间倒序；
- 支持按年份或月份分组；
- 卡片显示标题、摘要、标签、日期和封面；
- 无内容时显示空状态；
- V0.1 使用简单分页。

### 5.4 内容详情 `/entry/[slug]`

普通心得、随笔和理解共用详情布局，包含类型、标题、摘要、发布时间、最后修改时间、标签、封面和正文。

### 5.5 项目详情 `/project/[slug]`

包含名称、一句话介绍、封面、项目状态、开始日期、最后更新时间、技术栈、可选 GitHub 地址、正文、图片画廊和下一步计划。

### 5.6 关于 `/about`

包含我是谁、当前方向、正在做的事、为什么建立网站、联系方式和内容原则。

## 6. 私有端页面

### 6.1 登录 `/login`

- 只提供登录，不提供公开注册；
- 支持邮箱和密码；
- 登录成功跳转 `/studio`；
- 已登录访问登录页时跳转后台。

### 6.2 后台首页 `/studio`

包含新建内容、四种类型快捷入口、草稿数、已发布数、项目数、最近修改和最近发布。

### 6.3 内容管理 `/studio/entries`

支持标题搜索、类型筛选、状态筛选、标签筛选、按更新时间排序、编辑、发布/撤回、复制和删除到回收站。

### 6.4 新建 `/studio/new`

先选择内容类型，再选择对应模板或空白内容。

### 6.5 编辑 `/studio/edit/[id]`

字段：标题、Slug、类型、摘要、封面、标签、正文、是否精选、项目元数据、保存草稿、发布、撤回和删除。

必须显示保存状态、最后保存时间、创建时间、发布时间和未保存更改提醒。

### 6.6 回收站 `/studio/trash`

查看软删除内容并恢复。永久删除延后到 V0.2。

## 7. 编辑器需求

第一版工具栏：段落、H2、H3、加粗、斜体、删除线、有序列表、无序列表、引用、行内代码、代码块、分割线、链接、图片、撤销和重做。

第一版自定义组件：

### 提示块

用于重要说明、问题、提醒和结论。

### 认知变化块

字段：过去我认为、现在我认为、改变的原因。

### 图片块

字段：图片、alt、图片说明和宽度模式。

### 项目概览块

仅项目内容可用：项目状态、技术栈、时间范围和仓库链接。

不做完整 Notion 式拖拽块编辑器。

## 8. 图片上传

- 支持 JPG、PNG、WebP；
- 有大小限制；
- 显示上传进度；
- 失败可重试；
- 可填写 alt 和说明；
- 路径包含用户和内容 ID；
- V0.1 删除正文图片时不自动删除 Storage 文件，防止误删。

## 9. 日期和时间

系统自动维护：`created_at`、`updated_at`、`published_at`、可选 `last_published_at` 和 `deleted_at`。

公开端仅当更新时间明显晚于发布时间时显示“更新于”。

## 10. 自动保存

- 停止输入约 1.2 秒后保存；
- 显示正在保存、已保存和保存失败；
- 失败时保留本地内容；
- localStorage 作为临时兜底；
- 离开未保存页面时提醒；
- 发布前强制完成最后一次保存。

## 11. 搜索与标签

V0.1 搜索标题、摘要和纯文本正文；支持标签；后台和公开端均可搜索，公开搜索只返回已发布内容。

## 12. 精选

每条内容有 `featured` 和可选 `featured_order`。首页只展示少量精选内容。

## 13. 导出

V0.1 至少支持单篇 Markdown 和全部 JSON 导出。V0.2 再支持 Markdown ZIP 和图片打包。

## 14. 响应式与可访问性

- 375px 宽度可用；
- 后台支持手机快速写作；
- 键盘可访问；
- 焦点样式可见；
- 图片有 alt；
- 正文宽度适合中文阅读。

## 15. 非功能需求

- 未登录用户不能访问后台数据；
- 草稿不能通过 URL 或接口公开读取；
- 提交和上传失败不能导致正文丢失；
- 环境变量不得提交；
- 数据库变更使用 migration；
- 部署前通过 lint、typecheck 和 build。

## 16. V0.1 不做

多用户注册、评论、点赞、收藏、AI 写作、多语言、付费、实时协作、原生 App、复杂动画、高级统计、完整媒体库和所见即所得页面搭建器。

# 02 UI、交互与组件说明

## 1. 设计方向

关键词：克制、留白、清晰、现代、阅读优先、真实、安静。

可以借鉴高质量产品官网的层级、留白和排版，但不能复制任何品牌的代码、图片、文案或独特视觉资产。

## 2. 视觉原则

### 色彩

- 背景：接近白色或柔和浅灰；
- 主文字：接近黑色；
- 次级文字：中性灰；
- 强调色：仅一个；
- 深色模式放 V0.2。

避免大面积高饱和渐变、多强调色竞争、玻璃拟态堆叠、发光边框和赛博朋克视觉。

### 字体

- 中文正文优先系统中文字体；
- 标题层级清晰；
- 正文行高约 1.75；
- 正文最大宽度约 720–800px；
- 项目展示可适当加宽。

### 动效

允许 150–250ms 的淡入、按钮轻微位移和卡片边框变化。

禁止大面积视差、滚动绑架、长时间入场动画、自动播放声音和光标特效。

### `ImageUploader`

拖放、选择、进度、错误、预览、alt 和图片说明。

### `PublishPanel`

保存草稿、预览、发布、撤回、首页精选和发布时间。

## 10. 写作体验

新建内容时第一屏不展示大量配置：

```text
选择内容类型 → 输入标题 → 开始写作 → 需要时再展开摘要、封面和标签
```

后台默认专注模式，编辑区居中，侧栏可收起，自动保存，发布按钮清晰可见。

## 11. 页面状态

每个页面必须设计 loading、empty、error、success 和 unauthorized 状态，不允许只靠控制台报错。

# 03 技术架构

## 1. 推荐技术栈

```text
Next.js App Router
TypeScript
Tailwind CSS
Supabase Database
Supabase Auth
Supabase Storage
Tiptap
Zod
Vercel
```

## 2. 职责

- Next.js：公开页、后台页、动态路由、服务端读取、SEO 和接口。
- TypeScript：内容类型和表单约束。
- Tailwind：统一排版和组件样式。
- Supabase：Postgres、单用户认证和图片存储。
- Tiptap：可定制富文本编辑器。
- Zod：所有输入验证。
- Vercel：连接 GitHub 自动部署。

Tiptap 图片节点只负责显示，上传逻辑需要连接 Supabase Storage。

## 3. 总体架构

```text
浏览器
├── 公开端：Server Components 读取已发布内容
└── 私有后台
    ├── Supabase Auth 登录
    ├── Client Editor 交互
    └── Server Actions / Route Handlers 写入数据

Next.js
├── 页面与组件
├── 权限校验
├── 数据验证
├── 内容转换
└── SEO

Supabase
├── Auth
├── Postgres
└── Storage

Vercel
└── 构建、预览与生产部署
```

```text
│   ├── dates/
│   └── utils/
├── actions/
├── types/
└── styles/

supabase/
├── migrations/
└── seed.sql
```

## 5. 渲染策略

公开页面优先服务端渲染，只读取：

```text
status = published
AND deleted_at IS NULL
```

后台布局在服务端校验登录状态，编辑器本身是 Client Component。不能只靠前端隐藏后台。

## 6. 数据访问

建立独立函数：

```text
getPublishedEntries
getEntryBySlug
getAdminEntries
getEntryById
createEntry
updateEntry
publishEntry
archiveEntry
softDeleteEntry
restoreEntry
```

所有写操作必须：校验登录、Zod 校验、执行数据库操作、处理错误和重新验证缓存。

## 7. 编辑器内容格式

主内容保存为 Tiptap JSON：

```text
content_json: jsonb
```

同时维护：

```text
content_text: text
```

`content_text` 用于搜索，不作为富文本主数据。不要只保存 HTML。

## 8. Slug

- 新建时根据标题生成默认 slug；
- 允许管理员修改；
- 发布后保持稳定；
- 修改标题不自动改 slug；
- 数据库唯一；
- 查询不到返回 404；
- 中文 slug 可保留 Unicode，不为转写阻塞 V0.1。

## 9. 自动保存

```text
编辑 → 本地状态 → debounce 1200ms → 保存请求
→ 服务端验证 → 更新 updated_at → 返回保存时间 → 更新状态提示
```

保护措施：避免旧请求覆盖新内容、localStorage 快照、发布前强制保存。

## 10. 图片上传

```text
选择图片 → 客户端验证 → 生成路径 → 上传 Storage
→ 获得 URL/路径 → 插入图片节点 → 保存 alt 和 caption
```

建议路径：

```text
content-images/{adminUserId}/{entryId}/{uuid}.{ext}
```

V0.1 建议公开 bucket + 严格写入 RLS。

## 11. 缓存

公开列表可缓存；发布、撤回、删除和修改后必须 revalidate 对应路径；后台默认不长期缓存。

## 12. SEO

公开页面需要 title、description、canonical、Open Graph、sitemap、robots、发布时间和更新时间。后台与草稿禁止索引。

## 13. 环境变量

`.env.example` 至少包含：

```text
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
NEXT_PUBLIC_SITE_URL=
```

service role key 只能在服务端使用，`.env.local` 不提交。

## 14. 测试

单元测试：日期、slug、类型映射、Zod、Tiptap JSON 转纯文本。

集成测试：草稿、发布、撤回、上传、软删除和恢复。

E2E：

```text
管理员登录 → 新建心得 → 输入内容 → 上传图片 → 发布 → 公开页可见
```

## 15. 部署

GitHub 连接 Vercel；分支推送生成 Preview；生产分支合并后部署；配置 Supabase 环境变量和认证回调 URL；运行数据库 migration。

# 04 数据模型与安全

## 1. 核心表 `entries`

```text
id                uuid primary key
title             text not null
slug              text unique not null
type              text not null
status            text not null default 'draft'
summary           text
content_json      jsonb not null
content_text      text
cover_path        text
tags              text[] default '{}'
featured          boolean default false
featured_order    integer
metadata          jsonb default '{}'
created_by        uuid not null
created_at        timestamptz default now()
updated_at        timestamptz default now()
published_at      timestamptz
deleted_at        timestamptz
```

约束：

```text
type IN ('reflection', 'essay', 'project', 'understanding')
status IN ('draft', 'published', 'archived')
```

## 2. 管理员表 `app_admins`

```text
user_id           uuid primary key
created_at        timestamptz default now()
```

只保存允许管理网站的 Supabase Auth 用户 ID。

## 3. 项目元数据

项目可在 `metadata` 保存：

```json
{
  "projectStatus": "in_progress",
  "startDate": "2026-04-01",
  "endDate": null,
  "techStack": ["Arduino", "Python", "OpenCV"],
  "repositoryUrl": null,
  "demoUrl": null
}
```

第一版不为项目字段建立大量单独表。

## 4. 更新时间

数据库通过 trigger 自动维护 `updated_at`，不只依赖客户端。

## 5. 管理员判断

创建数据库函数 `is_app_admin()`，判断 `auth.uid()` 是否存在于 `app_admins.user_id`。

## 6. Entries RLS

### 匿名公开读取

仅允许：

```text
status = 'published'
AND deleted_at IS NULL
```

### 管理员

管理员可读取全部内容，并且只有管理员可以 insert、update 和 delete。

应用层默认使用软删除。

## 7. Storage

Bucket：`content-images`。

- 已发布内容图片可公开读取；
- 上传、更新和删除仅管理员；
- 路径必须以用户 ID 开头；
- 应用端限制 MIME 和文件大小。

公开 bucket 不代表任何人都能上传，上传仍需要 policy。

## 8. Auth

第一版只使用邮箱、密码和单一管理员账号，不提供注册页面。

管理员账号在 Supabase Dashboard 手动创建，再将 UID 写入 `app_admins`。

## 9. 路由保护

必须双重保护：

- 页面层：访问 `/studio/**` 时服务端检查 session；
- 数据层：RLS 拒绝未授权读写。

不能只使用前端 `if (user)` 隐藏按钮。

## 10. 内容安全

- Tiptap JSON 只允许白名单节点；
- 链接协议限制为 http、https、mailto；
- 不允许任意 script；
- 渲染 HTML 时使用安全转换；
- 上传文件检查 MIME；
- 用户输入使用 Zod；
- 错误不暴露数据库结构和密钥。

## 11. 备份

V0.1：单篇 Markdown、全部 JSON、migration 入库。

V0.2：Markdown ZIP、图片清单、定期备份和版本历史。

## 12. 删除策略

删除时设置：

```text
deleted_at = now()
```

恢复时设置：

```text
deleted_at = null
```

永久删除延后。

## 13. 隐私

公开端不得暴露管理员用户 ID、邮箱、草稿标题、私有路径、未发布项目和服务端错误堆栈。

## 14. Migration

所有表、触发器、函数和 RLS 使用 SQL migration 管理。真实管理员 UID 不写入公开仓库。

构建一个个人内容发布系统，而不是普通静态博客。

公开端展示：首页、内容空间、心得、随笔、项目、理解、内容详情、项目详情和关于我。

私有端提供：单用户登录、内容列表、新建/编辑/删除/恢复、草稿与发布、富文本编辑、图片上传、自动记录日期，以及首页精选控制。

## 技术约束

- 使用 Next.js App Router。
- 使用 TypeScript，开启严格模式。
- 使用 Tailwind CSS。
- 使用 Supabase Database、Auth 和 Storage。
- 使用 Tiptap 构建编辑器。
- 部署目标为 Vercel。
- 优先使用 Server Components；仅在交互确有需要时使用 Client Components。
- 不引入 Redux 等全局状态库，除非有明确必要。
- 不建立公开注册系统。
- 不在客户端暴露 Supabase service role key。
- 所有数据库表和 Storage bucket 必须配置 RLS 或等价的服务端权限保护。
- 不复制苹果官网代码、资产或品牌；只借鉴留白、层级、克制和阅读体验。
- 不使用夸张渐变、粒子、3D、自动播放背景视频或复杂滚动动画。
- 不虚构用户经历、项目成果、联系方式或技能等级。

## 开发规则

每次任务开始前：

1. 阅读相关 `docs/*.md`；
2. 检查当前仓库状态和已有实现；
3. 先给出简短实施计划；
4. 只修改当前阶段需要的内容。

每次任务结束前：

1. 运行 `npm run lint`；
2. 运行 `npm run typecheck`，若项目尚无该命令则补充；
3. 运行 `npm run build`；
4. 若有测试则运行测试；
5. 说明完成内容、修改文件、未完成内容和风险；
6. 不得声称未真实运行的检查已经通过。

## 代码质量

- 组件职责单一。
- 页面层不直接堆积大量业务逻辑。
- 数据访问统一放在 `lib/`、`services/` 或 server actions 中。
- 所有外部输入都需要验证，推荐使用 Zod。
- 统一处理错误、空状态和加载状态。
- 所有图片必须有 alt 文本。
- 所有交互必须支持键盘操作和可见焦点。
- 移动端宽度从 375px 起可用。
- 日期显示统一由一个工具函数或组件处理。
- Slug 创建后默认保持稳定，修改标题不应自动破坏旧链接。
- 删除默认采用软删除。

## 范围控制

V0.1 不做：多用户注册、评论/点赞/收藏、社交动态、多语言、AI 自动写作、实时协作、复杂数据分析、付费订阅、原生移动应用、Notion 级完整块编辑器和大型动画系统。

发现需求超出当前阶段时，记录到 `docs/08_BACKLOG.md`，不要擅自实现。
