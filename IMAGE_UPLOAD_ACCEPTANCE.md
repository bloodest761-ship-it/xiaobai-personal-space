# 图片上传与插入体验验收

## 原有问题

- 编辑器已能上传图片到 `content-images` bucket 并插入 `imageFigure` 节点，但单张限制为 5MB。
- 上传流程只有基础状态文字；上传期间没有双重防护，失败后也没有明确的重试入口。
- `alt` 和 `caption` 能在上传前写入节点，但选中已插入图片后不能继续编辑。

## 修改内容

- 图片选择限制为 JPG/JPEG、PNG、WEBP，拒绝 SVG 与其他文件；单张上限调整为 8MB。
- Storage 路径继续使用 `管理员 UID/文章 ID/UUID.扩展名`；不使用原始文件名作为路径。
- 上传中显示“正在上传图片……”，文件选择与重试入口禁用，并由 ref 再次拦截重复调用。
- 上传成功后才插入 `imageFigure`，失败不会写入损坏节点；失败保留已选文件并显示“重试上传”。
- 插入节点继续保存 `src`、`alt`、`caption`。若从编辑器光标启动选择器，则回填到该位置；没有编辑器焦点时插入文末。
- 点击编辑器图片会出现图片设置区，可修改替代文字和图片说明，或仅从文章 JSON 中移除图片节点。

## 上传流程

1. 在工具栏或“选择图片”按钮打开本地文件选择器。
2. 先校验 MIME 类型与 8MB 限制；不合规文件显示中文错误且不发起上传。
3. 合规文件立即显示上传状态；仅管理员经现有 Storage/RLS 策略可以写入 `content-images`。
4. 上传成功后获取实际公开 Storage 地址，插入含 `src`、`alt`、`caption` 的 Tiptap `imageFigure` 节点，并恢复编辑器焦点。
5. 节点变更会进入既有自动保存链路；刷新或编辑已有文章时按既有 JSON 解析回显。

## 错误处理

- 格式错误、超过 8MB、未登录、非管理员和网络异常都显示中文可理解提示。
- 上传中按钮禁用，且内部 `uploadInProgress` ref 阻止重复事件。
- 失败后不插入任何节点，并保留本次文件以供重试；正文内容不被改写。

## alt 和 caption

- 上传前可填写替代文字（提示建议填写）和可选图片说明。
- 选中已插入图片后，图片设置区可继续修改 `alt` 与 `caption`；修改会更新 Tiptap JSON 并触发正常自动保存。
- `src` 不提供编辑入口。公开页现有 `RichContent` 使用 `next/image` 渲染 `alt`，仅在 caption 非空时渲染 `figcaption`。

## 公开页面效果

- 现有公共渲染将图片限制为正文宽度内的 `width: 100%; height: auto`，并保持圆角、边框与编辑器一致。
- 全局 `img { max-width: 100%; }` 及移动端正文规则避免 375px 宽度产生横向溢出。
- 无效图片地址会被公开渲染的安全地址校验忽略；单张图片加载失败不会改变页面其他正文结构。

| 项目 | 结果 |
| --- | --- |
| lint | 已通过 |
| typecheck | 已通过 |

| 上传中重复点击 | 已静态检查：按钮禁用且有 `uploadInProgress` 双重拦截 |
| 模拟失败并重试 | 已静态检查：失败不插入节点、保留文件并显示重试入口 |

- 未使用管理员账号执行保存、刷新、发布和移动端视觉验收。
- 未删除任何 Storage 文件，也未修改数据库结构、RLS、登录流程或封面逻辑。
