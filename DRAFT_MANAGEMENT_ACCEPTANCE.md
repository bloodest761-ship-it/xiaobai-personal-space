# 草稿管理、删除确认、撤回与误操作保护验收

## 原有内容管理流程

原列表以服务端 `getAdminEntries` 查询，默认按 `updated_at` 倒序，且查询列表字段时不取正文。原有发布、撤回与删除分别存在于列表表单、编辑器和多个 Server Action；删除为基于 `deleted_at` 的软删除并进入既有回收站。

## 发现的问题

- 列表操作可直接提交，没有撤回/删除确认与逐条进行态。
- 已发布内容可从列表触发删除，旧实现不是本轮要求的永久删除保护。
- 生命周期缓存失效逻辑在两个 action 中重复。
- 列表包含全文搜索、类型和归档筛选，超出本轮轻量状态筛选范围。

## 新的列表结构

`/studio/entries` 服务端查询并按最近更新时间展示标题（空标题显示“无标题草稿”）、类型、文字状态、更新时间、创建时间、精选标记、编辑入口和状态操作。桌面端为多列列表，窄屏自动堆叠为可读卡片式行；列表不获取 `content_json` 或 `content_text`。

## 筛选方式

仅保留 URL 状态筛选：全部（无参数）、`?status=draft`、`?status=published`。非法参数回退到全部；筛选在服务器端完成，刷新和复制链接均能保留状态。

## 发布流程

草稿才能发布。编辑器先保存最新快照；服务端再次验证管理员、内容存在、当前为 draft、标题和正文非空，然后发布。进行中按钮禁用并显示进行态；成功后刷新 Studio、首页、`/space`、类型列表和详情路径及 tags。

## 撤回流程

仅已发布内容显示撤回。确认框明确说明撤回会从公开页消失但保留草稿；确认后服务端把状态改为 draft，不丢失内容、图片或 metadata，并刷新公开缓存。

## 删除保护

已发布内容不能直接删除，必须先撤回。仅草稿可打开永久删除确认框；必须输入“删除”才可提交。服务端再次以 `status=draft` 和未软删除条件执行永久删除，失败时列表项保留；不会自动删除 Supabase Storage 图片。

## 确认组件

`ConfirmActionDialog` 为轻量复用组件，支持说明、危险态、文字确认、操作中/错误态、Esc（提交中除外）、Tab 焦点约束与关闭后回到触发按钮。

## 缓存失效

`revalidateEntryViews` 集中刷新 `public-content`、`home`、对应类型和详情 tag，以及 `/`、`/space`、`/space/[type]`、详情页与 Studio 路径；没有改成全站动态渲染。

## 本地备份处理

永久删除成功后客户端才清理 `xiaobai-draft:<id>`；失败时保留。撤回不清理；编辑器发布成功仍将备份标记为已同步。列表不读取备份正文。

## 空状态与错误状态

加载沿用 Studio loading 骨架；全部、草稿和已发布筛选分别有明确空态。服务端返回中文可理解错误，客户端显示就地反馈且不暴露 Supabase 原始错误。

## 后续风险

仓库既有软删除/回收站代码为历史功能，本轮没有扩展。生产环境应确认 RLS 删除策略允许管理员对 draft 执行永久删除，同时继续禁止客户端绕过权限。
